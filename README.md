# Анализ музыкальных треков Spotify

Итоговый проект по дисциплине **«Анализ данных на Python»**.

Анализ ~114 000 треков Spotify с аудио-характеристиками (танцевальность,
энергичность, валентность и др.), обогащённых метаданными артистов через
внешний API. Цель — найти, какие характеристики связаны с популярностью
трека, и как различаются жанры.

## Структура

```
spotify_analysis/
├── notebook.ipynb        # основной ноутбук (5 блоков)
├── build_notebook.py     # генератор ноутбука из ячеек
├── README.md
├── requirements.txt
├── .env.example          # шаблон ключей Spotify
├── data/
│   ├── raw/dataset.csv   # исходный датасет
│   └── processed/        # обогащённые/очищенные данные (кэш)
└── src/
    └── enrich.py         # асинхронное обогащение через API
```

## Данные

**Источник:** [Spotify Tracks Dataset](https://huggingface.co/datasets/maharshipandya/spotify-tracks-dataset)
(автор maharshipandya, зеркало на HuggingFace без ключей). 114 000 треков,
114 жанров, 20 аудио- и мета-признаков.

Датасет содержит аудио-фичи, но не метаданные артиста, поэтому он
**обогащается через API** (популярность артиста, число подписчиков, жанры).

## Как запустить

```bash
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt

# (опционально) ключи Spotify для обогащения через Spotify API:
cp .env.example .env   # вписать SPOTIFY_CLIENT_ID / SPOTIFY_CLIENT_SECRET
# без ключей обогащение автоматически идёт через MusicBrainz (без ключа)

jupyter lab notebook.ipynb   # Run All
```

Если меняешь содержимое ноутбука — правь `build_notebook.py` и пересобирай:
`python build_notebook.py`.

## Технические усложнения (продвинутая часть)

1. **Асинхронный сбор данных** — обогащение артистов через `aiohttp` +
   `asyncio.gather` с семафором для rate-limit и обработкой HTTP 429
   (`src/enrich.py`).
2. **Оформление как портфолио** — структура репозитория с `README.md`,
   `requirements.txt`, выделенным модулем и кэшированием.

### Вероятные вопросы и ответы

**Блок 1 (Окулов М.А.):**
- *Почему обогащение через API, а не просто .csv?* — Готовый csv без
  обработки = 0 баллов за сбор. Обогащение метаданными артиста делает
  сбор «сложным».
- *Зачем asyncio, а не цикл с requests?* — Сетевые запросы ждут ответа;
  async позволяет слать их параллельно, ускоряя в N раз. Семафор
  ограничивает параллелизм, чтобы не словить rate-limit.
- *Что такое 429 и Retry-After?* — HTTP-код «слишком много запросов»;
  сервер в заголовке Retry-After говорит, сколько ждать.

**Блок 2 (Пирогов Е.А.):**
- *Почему медиана для пропусков, а не среднее?* — Медиана устойчива
  к выбросам, среднее их «тянет».
- *Что такое IQR-выбросы?* — Межквартильный размах: значения за
  пределами Q1−1.5·IQR … Q3+1.5·IQR считаются выбросами.
- *Почему винзоризация, а не удаление?* — Не теряем строки, но
  ограничиваем влияние экстремумов на статистики.
- *Зачем две версии df?* — Один track_id встречается в разных жанрах:
  `df` для анализа по жанрам, `df_unique` — по уникальным трекам.

**Блоки 3-4 (Игонин А.Е.):**
- *Почему Спирмен/Манн-Уитни, а не Пирсон/t-test?* — Популярность
  распределена не нормально; непараметрические тесты не требуют
  нормальности.
- *Что значит p-value < 0.05?* — Вероятность увидеть такой результат
  случайно мала; отвергаем нулевую гипотезу «связи/различия нет».
- *Корреляция = причинность?* — Нет. Мы фиксируем связь, не причину.
